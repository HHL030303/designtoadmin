# 设计交付管理系统 · 流程平台方案 v1.0

> 基于《课件全流程管理系统_产品方案文档 v1.0》重构  
> 目标：在现有系统基础上，建设一套面向设计交付的流程流转平台  
> 约束：系统外虚拟角色不纳入系统，所有审核/确认默认在线下或企微完成，系统只承接结果确认与流程推进

---

## 一、方案定位

这套平台不是重新设计业务流程，而是将现有文档中的课件生产链路，抽象为一套可配置、可追踪、可预警、可归档的流程管理平台。

平台核心只做四件事：

1. 承接任务录入与节点拆解
2. 驱动各系统内角色之间的任务流转
3. 沉淀每个阶段的交付物、时间节点和责任人
4. 在不介入审核过程的前提下，保证状态可追踪、结果可归档、版本可追溯

---

## 二、系统边界重定义

### 2.1 纳入系统的角色

| 角色 | 平台定位 | 核心动作 |
|------|---------|---------|
| 计划员 | 流程发起人/版本发起人 | 新建任务、设定节点、发起售后、发起迭代 |
| 教研老师 | 教研交付人 | 上传教研资料、确认教研完成 |
| 设计统筹 | 流程调度人/入库确认人 | 风格稿派单、内页派单、确认入库 |
| 风格稿设计师 | 风格稿执行人 | 接收任务、下载资料、上传风格稿 |
| 内页设计师 | 内页执行人 | 接收任务、下载资料、上传内页成品 |
| 售前人员 | 进度查看人/售后提报人 | 查看状态、发起售后工单 |
| 管理员 | 平台管理人 | 用户权限、流程配置、日志查看 |

### 2.2 不纳入系统的角色

以下角色全部视为系统外协同角色，不在平台内建账号、待办、审批节点：

- 教研审核员
- 风格稿审核员
- 内页审核员
- 最终质检员

处理原则：

- 审核动作在线下或企微完成
- 系统不记录“审核中/审核驳回/审核通过”的过程态
- 系统只记录可落地的结果态，例如“已上传风格稿”“已上传内页成品”“确认入库”
- 若线下确认未通过，由执行人在线下修改后重新上传，系统以最新一次上传结果为准

## 三、平台设计目标

相比原始文档，这套平台建议强化为“流程中心”，而不只是“状态记录工具”。

### 3.1 业务目标

- 统一设计交付的任务入口、派单入口、上传入口、归档入口
- 让每个系统内角色只看到自己该做的动作
- 用状态机驱动而非人工记忆驱动流程推进
- 支持主流程、售后、迭代三类业务在同一平台内闭环

### 3.2 管理目标

- 每个节点有责任人、截止时间、产物、完成时间
- 每次流转都有操作记录
- 每个版本都可回溯来源、变更原因、交付物
- 逾期自动预警，不依赖人工盯表

---

## 四、核心业务对象

建议平台统一围绕以下 5 个对象设计。

### 4.1 课件主单

平台中的主对象，承载课件基础信息、当前状态、当前版本、总负责人、总时间节点。

### 4.2 阶段任务

用于承接主流程中的每一个执行节点，建议固定为 4 类：

- 教研任务
- 风格稿任务
- 内页任务
- 入库任务

每个阶段任务都具备：

- 任务类型
- 当前负责人
- 协作人
- 截止时间
- 实际完成时间
- 上传附件
- 任务状态

### 4.3 子任务单

用于承接非主流程业务：

- 售后单
- 迭代单

子任务单必须挂靠某个课件主单，不能脱离主单独立存在。

### 4.4 交付物

平台统一管理三类交付物：

- 教研资料包
- 风格稿成品
- 内页成品

每类交付物都需记录：

- 上传人
- 上传时间
- 文件清单
- 规范校验结果
- 所属版本

### 4.5 流转日志

用于还原整个流程过程，建议记录：

- 谁在什么时间做了什么动作
- 状态从什么变成什么
- 附带说明是什么
- 是否系统自动触发

---

## 五、统一流程模型

建议将平台抽象成“主流程 + 分支流程”的模型。

### 5.1 主流程

适用于全新订单，也作为迭代订单的默认模板。

```text
新建任务
→ 教研中
→ 待风格稿派单
→ 风格稿制作中
→ 待内页派单
→ 内页制作中
→ 待入库确认
→ 打包中
→ 已归档
```

### 5.2 分支流程

- 售后流程：挂靠已归档版本，走轻流程，完成后覆盖生成小版本
- 迭代流程：挂靠已归档版本，按勾选节点重走完整或半完整流程，完成后生成大版本

这样平台只有一套主状态机，但允许不同单据以不同方式进入和退出。

---

## 六、角色间状态流转设计

这是这套平台的核心。因为系统外审核角色被排除，所以状态流转必须只围绕系统内角色的“可执行动作”设计。

### 6.1 主流程状态机

| 状态 | 责任角色 | 触发动作 | 下一状态 |
|------|---------|---------|---------|
| 草稿 | 计划员 | 新建课件并提交 | 教研中 |
| 教研中 | 教研老师 | 上传教研资料并确认完成 | 待风格稿派单 |
| 待风格稿派单 | 设计统筹 | 指派风格稿设计师并设置截止时间 | 风格稿制作中 |
| 风格稿制作中 | 风格稿设计师 | 上传风格稿成品 | 待内页派单 |
| 待内页派单 | 设计统筹 | 指派内页设计师并设置截止时间 | 内页制作中 |
| 内页制作中 | 内页设计师 | 上传内页成品且系统校验通过 | 待入库确认 |
| 待入库确认 | 设计统筹 | 查看完整性报告并确认入库 | 打包中 |
| 打包中 | 系统 | 打包并写入成品库成功 | 已归档 |
| 打包失败 | 系统/设计统筹 | 重新触发打包 | 打包中 / 待入库确认 |
| 已归档 | 只读 | 发起售后或发起迭代 | 售后处理中 / 迭代中 |

### 6.2 关键原则

- “上传即视为线下已确认”，不增加审批状态
- 每个状态只对应一个明确责任角色
- 每个状态只暴露一个主动作按钮，避免角色误操作
- 系统自动态只出现在“校验中/打包中/打包失败”这类机器节点

### 6.3 建议增加的异常状态

原文档里对异常流转描述较少，平台化后建议补足：

| 状态 | 含义 | 说明 |
|------|------|------|
| 已退回修改 | 线下未确认通过，需要重新上传 | 不做审批流，只做人工退回标记 |
| 已挂起 | 任务暂停 | 例如资料缺失、需求待确认 |
| 已取消 | 任务终止 | 保留日志，不参与统计 |
| 打包失败 | 自动打包异常 | 便于运维和设计统筹排查 |

其中“已退回修改”不是审批节点，而是设计统筹或计划员对当前阶段任务做的协同标记，帮助团队识别当前任务仍需继续处理。

## 七、按角色拆分的工作台设计

平台不建议所有人都看同一张大表，而应按角色给到不同工作台。

### 7.1 计划员工作台

核心内容：

- 新建课件
- 查看全部课件进度
- 查看逾期课件
- 发起售后
- 发起迭代
- 查看版本记录

建议重点视图：

- 我创建的
- 本周待完成教研
- 即将逾期
- 已归档可迭代

### 7.2 教研老师工作台

核心内容：

- 待处理教研任务
- 已完成教研任务
- 上传教研附件
- 确认教研完成

页面动作尽量简化为：

- 下载任务资料
- 上传附件
- 确认完成

### 7.3 设计统筹工作台

这是整个平台最重要的调度工作台，建议拆成三个栏目：

- 待风格稿派单
- 待内页派单
- 待入库确认

每张卡片至少显示：

- 课件名称
- 当前版本
- 当前阶段
- 当前责任人
- 截止时间
- 逾期标识
- 附件完整性结果

### 7.4 设计师工作台

风格稿设计师与内页设计师共用一套工作台，通过任务类型区分。

核心内容：

- 我的待办
- 我参与的任务
- 已逾期任务
- 已完成任务

单任务详情页建议显示：

- 基础信息
- 教研附件下载区
- 文件命名规范提示
- 上传区
- 上传历史

### 7.5 售前工作台

只做轻量化：

- 查看课件当前状态
- 搜索课件
- 发起售后工单
- 查看售后处理结果

## 八、流程动作设计

建议平台中的所有流转都统一为“动作驱动”，而不是自由编辑状态。

### 8.1 角色动作清单

| 角色 | 动作 |
|------|------|
| 计划员 | 新建任务、编辑任务、发起迭代、发起售后、挂起、取消 |
| 教研老师 | 上传教研资料、确认教研完成 |
| 设计统筹 | 风格稿派单、内页派单、退回修改、确认入库、重试打包 |
| 风格稿设计师 | 接收任务、上传风格稿、重新上传 |
| 内页设计师 | 接收任务、上传内页、重新上传 |
| 系统 | 自动校验、自动打包、自动归档、自动预警 |

### 8.2 按钮权限建议

每个状态只开放最少按钮：

- 教研中：`上传资料`、`确认教研完成`
- 待风格稿派单：`风格稿派单`
- 风格稿制作中：`上传风格稿`
- 待内页派单：`内页派单`
- 内页制作中：`上传内页成品`
- 待入库确认：`查看完整性报告`、`确认入库`
- 打包失败：`查看失败原因`、`重新打包`

这样前端实现会更稳定，使用成本也更低。

## 九、售后与迭代平台化设计

### 9.1 售后流程

售后不建议继续作为“备注型记录”，建议独立成单，但挂靠原版本。

流程建议：

```text
已归档课件
→ 发起售后单
→ 待接单
→ 处理中
→ 已上传修正版本
→ 自动归档
→ 工单关闭
```

建议特点：

- 售后单独编号
- 必须关联原课件和原版本
- 自动生成新小版本号，如 `v1.1`
- 保留修改原因、责任归因、修正记录

### 9.2 迭代流程

迭代建议继续复用主流程引擎，但增加“跳过节点”的能力。

示例：

- 只改内容，不改设计：跳过风格稿、内页
- 改内页，不改教研：跳过教研
- 全量升级：一个节点都不跳

流程建议：

```text
已归档课件
→ 发起迭代单
→ 选择需执行节点
→ 自动生成阶段任务
→ 按主流程推进
→ 自动归档
→ 生成大版本号
```

版本规则建议：

- 售后：`v1.0 -> v1.1`
- 迭代：`v1 -> v2`

若希望统一，也可以全部规范成：

- 售后：`v1.0 -> v1.1`
- 迭代：`v1.0 -> v2.0`

这个表达更适合系统实现与用户理解。

## 十一、预警与看板设计

这套平台的价值不只在“流程走完”，还在于管理可视化。

### 11.1 预警规则

- 当前时间超过阶段截止时间且未完成，标记逾期
- 上传校验失败，标记待修正
- 打包失败，标记异常
- 挂起超过设定时长，标记长期挂起

### 11.2 看板指标

建议首页提供以下指标：

- 本月新建课件数
- 当前处理中课件数
- 待风格稿派单数
- 待内页派单数
- 待入库确认数
- 逾期任务数
- 本月已归档数
- 设计师完成量排行
- 售后率/返修率

## 十二、模块结构建议

建议平台按下面 8 个一级模块组织。

| 模块 | 说明 |
|------|------|
| 课件主单管理 | 新建、编辑、搜索、筛选、详情 |
| 流程任务中心 | 教研、风格稿、内页、入库节点驱动 |
| 设计师任务台 | 我的待办、我的上传、我的逾期 |
| 交付物管理 | 各阶段附件、版本附件、压缩包归档 |
| 售后中心 | 售后提报、接单、修正、关闭 |
| 迭代中心 | 迭代发起、节点勾选、版本生成 |
| 预警与统计 | 逾期、失败、挂起、月度统计 |
| 系统配置中心 | 流程配置、文件规范模板、权限、日志 |

## 十三、推荐的实施策略

如果你们是“基于当前系统扩展”，不建议一次性做成很重的平台，建议分三期。

### 第一期：先把主流程跑通

范围：

- 课件主单
- 教研上传
- 风格稿派单与上传
- 内页派单与上传
- 入库确认
- 自动打包
- 基础状态机

目标：

- 从 Excel/企微记录，切换到系统内统一流转

### 第二期：补齐管理能力

范围：

- 逾期预警
- 设计师任务台
- 流转日志
- 统计与导出
- 文件规范模板配置

目标：

- 从“能用”提升到“可管理”

### 第三期：补齐版本能力

范围：

- 售后中心
- 迭代中心
- 历史版本对比
- 成品库接口深度联动

目标：

- 从“流程记录系统”升级为“版本化交付平台”

## 十五、结论

基于你当前文档，这套“设计交付管理系统”最合适的产品定位，不是审批系统，也不是单纯附件系统，而是：

**一套围绕课件版本交付的流程流转平台。**

它的关键不是把线下审核搬进系统，而是：

- 用系统内角色串起每个可执行节点
- 用清晰状态机替代口头推进
- 用交付物和版本记录承接结果
- 用预警、日志、统计支撑日常管理

在这个前提下，虚拟角色继续留在线下协同，系统只接结果，反而更轻、更稳，也更符合你们当前的实际工作方式。
